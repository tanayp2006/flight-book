PROJECT OVERVIEW
================================================================================
This is a Next.js App Router flight booking application using:
- Next.js 16 + TypeScript
- Prisma ORM + PostgreSQL (Neon)
- Better Auth for authentication/session
- Tailwind CSS for UI

The app currently includes:
- Role-based navigation and route protection
- AIRPORT_MANAGER-only admin routes and admin APIs
- User search/discovery flow on /flights with autocomplete and filters
- Booking details route (/bookings/[id]) to keep click flow valid

================================================================================
DATABASE AND MODELS
================================================================================
Prisma models:
- User
- Session
- Account
- Verification
- Airport
- Airline
- Airplane
- Flight

Enums:
- UserRole: USER | AIRPORT_MANAGER | ADMIN
- FlightStatus: SCHEDULED | DELAYED | BOARDING | CANCELLED

Current access rule in code:
- Admin behavior is granted only to AIRPORT_MANAGER.
- ADMIN enum still exists in schema but is not considered admin by current guards.

Seeding:
- prisma/seed.ts populates airports, airlines, airplanes, and flights for testing.

================================================================================
AUTHENTICATION AND ROLE RESOLUTION
================================================================================
Authentication:
- Better Auth wired through app/api/auth/[...all]/route.ts
- Session retrieved via auth.api.getSession

Role resolution:
- Role fetched from DB on protected server routes.
- Middleware role source endpoint:
  - GET /api/auth/role

Shared helper:
- lib/route-auth.ts
  - getSessionAndRole(request)
  - isAdminRole(role) -> role === AIRPORT_MANAGER

================================================================================
ROUTES AND ACCESS CONTROL
================================================================================
Public:
- /
- /login

Utility:
- /redirect
  - Post-login redirection based on role and optional target/from query params.

Dashboard:
- /dashboard
- /dashboard/admin
- /dashboard/user

Search and booking:
- /flights
- /bookings/[id]

APIs:
- /api/auth/[...all]
- /api/auth/role
- /api/flights/search
- /api/admin/flights
- /api/admin/flights/[id]
- /api/admin/airlines
- /api/admin/airports

Middleware (middleware.ts):
- Protects /dashboard/*, /flights/*, /bookings/*, /api/admin/*
- If unauthenticated:
  - redirects page routes to /login
  - returns 401 for protected API routes
- If non-admin on admin routes:
  - redirects page routes to /dashboard/user
  - returns 403 for admin APIs
- If AIRPORT_MANAGER visits user-protected surfaces:
  - redirects to /dashboard/admin

USER SEARCH AND DISCOVERY (PHASE 2)
================================================================================
Main search page:
- app/flights/page.tsx

Hero search and filters:
- components/FlightHeroSearch.tsx

Implemented search inputs:
- Source airport (autocomplete via datalist)
- Destination airport (autocomplete via datalist)
- Date
- Airline (autocomplete via datalist)

Query and validation handling:
- Date strict parse (YYYY-MM-DD) with UTC day boundaries
- Price parsing supports minPrice, maxPrice, and compact priceRange

- min/max auto-corrected when reversed

Search API:
- app/api/flights/search/route.ts
- Supported params:
  - sourceAirportId
  - destinationAirportId
  - airlineId
  - date
  - minPrice
  - maxPrice
  - priceRange

Results UI:
- components/FlightResults.tsx
- Displays route, timing, duration, stop estimate, airline visual, price, capacity
- Book action links to /bookings/[id]
